#+STARTUP: overview

Start the development environment with:

#+begin_src bash
./bin/env-start
#+end_src 

Once started, an editor opens this very file. Its basic commands are:
- Move the point using the arrows.
- Place the point on a section and press <tab> to unfold it.
- Click on a link to view references to it.
- A link that starts with "⊙" defines a location, /i.e./ it says "here".
- A link that starts with "→" defines a reference to a location, /i.e./ it says "got to the
  referenced location".
- Quit the editor using C-x C-c.
- It means: press on <control> and "x", then: press on <control> and "c".

To understand /what/ ~./bin/env-start~ is supposed to do, you may consult its /spec/, /i.e./
/specification/, which is referenced from its source code. See: [[ref:b80c22c9-2eed-4eea-a469-136e7666ea96]]. To understand /how/ it
does what it is supposed to do, read the source code.

To introduce a new piece of code or data, we first write its specification.

** env-start
:PROPERTIES:
:ID:       d8afe12e-f756-4c33-a7dd-f380004f91bf
:END:

Premises
- git is installed
- ssh-add is installed
- guix is installed
- ./env-start
  - [[ref:b80c22c9-2eed-4eea-a469-136e7666ea96]]

Conclusion
- A development environment has been started.
- The current bash session occurs within a container.
- The session is configured for development.
  - It's init file is: [[ref:7cab5b80-5358-4093-b9fa-9d23665ab64a]]
- It is isolated from the host except for:
  - network access,
  - Guix daemon access to share build capabilities with the host,
  - read access to /etc/guix to sign builds,
  - access to the underlying ssh agent in order to push commits to the remote repository,
  - /tmp is whared with the host,
  - can lunch graphical applications (Wayland),
  - Host Guix archive keys are accessible in order to send builds to external resources /e.g./ VPS.
  - Host SSH agent can be used in order to connect to external resources.
- A set of packages are installed.
  - [[ref:ddced880-d052-4730-afd1-51fa39d7404c]]
- Installed packages have a fixed version.
  - [[ref:de35545c-ebb7-41db-b7c7-c8fcff0422f2]].
- The README is displayed using the default editor

** VM spec
:PROPERTIES:
:ID:       9952e0a0-6fe0-40cb-97fd-1d56074ab2a5
:END:

*Guest Virtual Machine Specifications*

- Architecture :: x86_64 virtual machine
- Chipset & Acceleration :: Modern Q35 PCIe-based machine model with full KVM hardware virtualization acceleration (near-native performance)
- CPU Model :: Exact host CPU feature passthrough (complete CPUID match including AVX-512 and all host extensions; zero emulation overhead for supported instructions)
- Memory Allocation :: Exactly 8 GiB of guest physical RAM
- CPU Topology :: Exactly 4 virtual CPUs visible inside the guest (configured as 1 socket with 1 thread per core)
- Storage Backend :: QCOW2 disk image; all guest writes are collected into a temporary overlay that is automatically discarded on VM exit (original image remains completely unchanged)
- Block Device :: Paravirtualized virtio-blk-pci device connected to the QCOW2 image (high-throughput, low-overhead I/O)
- Networking Backend :: User-mode (SLIRP/NAT) networking with three explicit TCP port forwards from host to guest:
  - SSH: host port → guest port 22
  - HTTP: host port → guest port 80
  - HTTPS: host port → guest port 443
- Network Device :: Paravirtualized virtio-net-pci interface with checksum offload and multi-queue support
- Control Interface :: QEMU Machine Protocol (QMP) monitor exposed on a Unix domain socket; external tools can connect and issue JSON commands (query-status, system_powerdown, etc.)

*Host Process & Management Specifications*

- Execution Context :: Runs as a transient systemd user unit (named after the supplied unit variable)
- Environment Inheritance :: Full user environment, cgroups, and resource limits
- Lifecycle Behavior :: Automatic cleanup on user logout unless remain-after-exit is explicitly requested
