This file describes this directory. Its intended use is to develop a Phoenix application
with a Lovable generated frontend.

The development occurs in a reproducible container in order to maximize team efficiency as
we all share the exact same development environment, irrespective of our particular setup.

This environment attempt to be documented enough so that AI can easily be used accurately
within the boundaries of the container.

Given this intented use, some parts of the directory are marked as PUBLIC as they should
be directly used in order to reach the stated goal. Others are private by default as they
exist only to support the public parts.

*** VM spec
:PROPERTIES:
:ID:       9952e0a0-6fe0-40cb-97fd-1d56074ab2a5
:END:

*Guest Virtual Machine Specifications*

- Architecture :: x86_64 virtual machine
- Chipset & Acceleration :: Modern Q35 PCIe-based machine model with full KVM hardware virtualization acceleration (near-native performance)
- CPU Model :: Exact host CPU feature passthrough (complete CPUID match including AVX-512 and all host extensions; zero emulation overhead for supported instructions)
- Memory Allocation :: Exactly 8 GiB of guest physical RAM
- CPU Topology :: Exactly 4 virtual CPUs visible inside the guest (configured as 1 socket with 1 thread per core)
- Storage Backend :: QCOW2 disk image; all guest writes are collected into a temporary overlay that is automatically discarded on VM exit (original image remains completely unchanged)
- Block Device :: Paravirtualized virtio-blk-pci device connected to the QCOW2 image (high-throughput, low-overhead I/O)
- Networking Backend :: User-mode (SLIRP/NAT) networking with three explicit TCP port forwards from host to guest:
  - SSH: host port → guest port 22
  - HTTP: host port → guest port 80
  - HTTPS: host port → guest port 443
- Network Device :: Paravirtualized virtio-net-pci interface with checksum offload and multi-queue support
- Control Interface :: QEMU Machine Protocol (QMP) monitor exposed on a Unix domain socket; external tools can connect and issue JSON commands (query-status, system_powerdown, etc.)

*Host Process & Management Specifications*

- Execution Context :: Runs as a transient systemd user unit (named after the supplied unit variable)
- Environment Inheritance :: Full user environment, cgroups, and resource limits
- Lifecycle Behavior :: Automatic cleanup on user logout unless remain-after-exit is explicitly requested
