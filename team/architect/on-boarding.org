** Development environment
*** Install Guix

To make the development environment fully reproducible, we use Guix. Install Guix
following these guides:

1. https://guix.gnu.org/manual/devel/en/guix.html#Binary-Installation
2. https://guix.gnu.org/manual/devel/en/guix.html#Name-Service-Switch-1
3. https://guix.gnu.org/manual/1.5.0/en/html_node/Upgrading-Guix.html

It is a good idea to update the guix daemon using an automatic job. If using systemd,
then why not:

=/etc/systemd/system/guix-update.service=

#+begin_src conf
[Unit]
Description=Update Guix Daemon
Wants=network-online.target
After=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/bin/guix pull
ExecStart=/usr/bin/systemctl restart guix-daemon.service
#+end_src

=/etc/systemd/system/guix-update.timer=

#+begin_src conf
[Unit]
Description=Run Guix daemon update weekly

[Timer]
OnCalendar=weekly
Persistent=true
RandomizedDelaySec=1h

[Install]
WantedBy=timers.target
#+end_src

Then:

#+begin_src bash :eval no
systemctl daemon-reload
systemctl enable --now guix-update.timer
#+end_src

*** Secrets

Your secrets are stored encrypted in ~.env.gpg~ at the root of the repository.
~make dev-env~ decrypts this file and loads its variables into the environment
automatically.

**** Creating ~.env.gpg~ for the first time

Write a plain ~.env~ file:

#+begin_src bash :eval no
XAI_API_KEY=xai-YOUR-KEY-HERE
#+end_src

Encrypt it with your GPG key:

#+begin_src bash :eval no
gpg --encrypt --recipient your@email.com .env
#+end_src

This produces ~.env.gpg~. Delete the plaintext file immediately:

#+begin_src bash :eval no
# Note: shred is ineffective on SSDs and copy-on-write filesystems (btrfs, zfs) — on
# those, avoiding writing the plaintext to disk at all.
shred -u .env
#+end_src

**** Editing existing secrets

Decrypt, edit, then re-encrypt:

#+begin_src bash :eval no
gpg --decrypt .env.gpg > .env
# edit .env
gpg --encrypt --recipient your@email.com .env
rm .env
#+end_src

*** Start the application

Start the development environment:

#+begin_src bash :eval no
make dev-env
#+end_src

To verify everything works, clone a test Lovable project under =frontend/=:

#+begin_src bash :eval no
git clone https://github.com/phf-1/my-lovely-tasks frontend
#+end_src

Then start a REPL:

#+begin_src bash :eval no
,start-repl
#+end_src

The application should now be live.

** Authenticated commits
*** Generate a keypair

If you do not already have one, generate a GPG keypair. This is used to sign and
identify your commits:

#+begin_src bash :eval no
gpg --batch --full-generate-key <<EOF
Key-Type: eddsa
Key-Curve: ed25519
Key-Usage: cert
Subkey-Type: eddsa
Subkey-Curve: ed25519
Subkey-Usage: sign
Name-Real: YOUR NAME
Name-Email: YOUR@EMAIL.COM
Expire-Date: 2y
Passphrase: YOUR-STRONG-PASSPHRASE-HERE
%commit
EOF
#+end_src

Add an encryption subkey:

#+begin_src bash :eval no
# Get your key fingerprint
gpg --list-keys YOUR@EMAIL.COM

# Add the cv25519 encryption subkey
gpg --quick-add-key YOURFINGERPRINT cv25519 encr 2y
#+end_src

Verify the result:

#+begin_src bash :eval no
gpg --list-keys --with-subkey-fingerprint YOUR@EMAIL.COM
#+end_src

*** Configure Git to sign commits

Tell Git to sign commits and tags with your key:

#+begin_src bash :eval no
# Note: you may want to use --global so that it is applied to all git projects.
git config user.signingkey 0x… # your signing key handle
git config commit.gpgsign true
git config tag.gpgsign true
#+end_src

Verify:

#+begin_src bash :eval no
git config --list | grep sign
#+end_src

*** Add your public key to your profile

Export your public key:

#+begin_src bash :eval no
gpg --armor --export your@email.com
#+end_src

Then:
- Copy [[ref:1446c391-2c77-4290-be29-b4fa76a751c9][this developer profile]] and its associated directory, updating names and IDs as appropriate.
- Complete your identification. Example: [[ref:73a7c491-ef86-4523-bfad-f024fef4fa07][Identification]].

Send a pull request.

*** Key signing by the architect

Once your PR is merged, the [[ref:87e740ec-6036-4154-968d-625196f90ee4][architect]] will sign your public key and configure the
platform to accept commits signed by you. Your contributions will then be
authenticated by a chain of trust.
